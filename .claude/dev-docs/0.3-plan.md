## §1 · Object model & terminology

Naming convention adopted from Hermes's proposal on 2026-04-17 after reconciling with the bot's first cut and cross-checking research.

| Term | Defined as | Notes |
|---|---|---|
| `agent` | Any model-driven actor in a scene. | System-under-test, reviewer, adversary, or simulated user. Identity = role + tool access + prompt. |
| `simulated user` | A degenerate agent whose role is "user". | Not a separate concept. Same orchestration path. Persona + LLM + optional MCP surface. |
| `turn` | One actor, one utterance. | Smallest conversational unit. User turn, agent turn, tool turn. |
| `round` | One bidirectional exchange between two actors. | `agent ↔ user` or `agent ↔ agent`. Composes multiple turns. |
| `step` | One action against env/tool + its observation. | Gym convention: `env.step(a) → (o, r, done, info)`. |
| `trajectory` | Full ordered trace of turns, rounds, and steps for one trial. | What the verifier reads and the publisher ships. |

Three orthogonal axes: **multi-turn × multi-round × multi-step**. Single-round multi-step = SWE-Bench. Multi-round single-step = chatbots. Multi-round & multi-step = ClawsBench, nanofirm.

**Sources:** OpenAI Gym (`step`); task-oriented dialogue literature, *Flipping the Dialogue* arXiv:2510.06552 (`turn`/`round`); MCP-sidecar pattern in `docs/multi-agent-sandbox-design.md` (simulated user).

**Consequence for metrics:** turns-per-role, rounds-to-resolution, steps-per-success, communication-overhead-vs-reward, round-level process reward.

|---|---|---|---|

## §A1 · Multi-agent, multi-round runtime

Docker Compose scene with N agent containers, an orchestrator with an A2A bus + mailbox + MCP router, a shared workspace volume, and pluggable mock-service sidecars.

**Three comms substrates:**
- **A2A** — peer agents. JSON-RPC 2.0 over HTTP/SSE/gRPC. Adopt rather than roll our own.
- **MCP** — agent ↔ tools. Existing in-tree; wire runtime end-to-end for 0.3.
- **Mailbox** — async / realistic. Small HTTP "email" service with delivery delays and inboxes. Shared implementation with the §A3 gmail sidecar.

**Build path:**
1. Lift `docs/multi-agent-sandbox-design.md` from draft. Compose file becomes the runtime definition.
2. Orchestrator container runs A2A endpoint + mailbox + MCP router in one binary.
3. Each agent container gets an `agent.yaml` — role, peers, tools.
4. Trajectory emits distinct `TrajectoryTurn`, `TrajectoryRound`, `TrajectoryStep` events.
5. Budget enforcement: per-trial round cap, per-agent token cap, scene timeout.

**Risks:** cost explosion (N × M), reproducibility under async + LLM non-determinism (record/replay companion feature), simulated-user placement in core vs ClawsBench.

## §A2 · Dense rewards + env snapshots *(RL core, ORS reward-signal-compatible in 0.3)*

**Reward-signal-first ORS interface.** ORS extends MCP semantics with RL primitives. For 0.3, benchflow should expose the reward/event surface over HTTP+SSE and keep the shape compatible with ORS concepts: Tools (action surface), Episodes (session until `finished`), Reward signals (streamed), with Tasks/splits/prompts completed only if they do not slow the release.

**Dense / process rewards wired end-to-end:**
- Agent SDK: `session.emit_process_reward(value, tag)`. Runner proxies to ORS `reward` event stream.
- Runner: writes `rewards.jsonl` alongside `reward.json`; also emits SSE to ORS-subscribed trainers.
- Verifier: returns a rubric (list of scored items), not just a scalar. Schema extension with backwards-compat shim.
- Aggregator: per-step, per-round, terminal reward surfaces.

**Env snapshots — cross-backend:**
- Docker: `docker commit` + `docker export` of overlay diff. Acceptable for single-agent, too slow for branching rollouts.
- Daytona: built-in `SnapshotState` API (`harbor/environments/daytona.py:21`). Target for fast rollout branching.

**Build path:** wire `emit_process_reward` through ACP + runner + storage → rubric verifier → Docker snapshot prototype → Daytona snapshot prototype → reference task `tasks/rl-cartpole-mock` (10-step env, shaped reward per step). Freeze the reward/event wire first; defer any non-essential ORS endpoints.

## §A3 · ClawsBench as benchflow-native environment

ClawsBench becomes a first-class benchflow environment family. Its mock services (gmail, gcal, gdocs, gdrive, slack) ship as benchflow-managed sidecars via `[[environment.services]]` in `task.toml`.

```toml
[[environment.services]]
name     = "gmail"
snapshot = "acme-corp-2026-03-01"
policy   = "fresh-per-trial"
```

**Guarantees:** reproducibility (per-trial fresh state, deterministic seeds), fidelity (real-ish APIs with delivery delays, rate limits, permission denials), composability (any task opts into any subset).

**Build path:** extract mock services into `benchflow-services/` → define schema + Compose boot → snapshots as tagged container images + `state.tar.zstd` in registry → 3 starter snapshots (*acme-corp*, *indie-dev*, *research-lab*) → CLI `bench environment snapshot create`.

## §A4 · Gym-like step API + observation/action specs *(RL core)*

Makes benchflow addressable by RL libraries (TRL, Verl). Current job-based model is wrong for RL — we need a live loop where the trainer drives `env.step(action)`.

```python
import benchflow as bf

env = bf.make("clawsbench/inbox-triage", agent="claude-agent-acp")
obs, info = env.reset()
done = False
while not done:
    action = policy(obs)
    obs, reward, done, info = env.step(action)
env.close()
```

**Observation / action spec in `task.toml`:**

```toml
[observation]
kind    = "tool_results"    # or "text", "structured", "pixels"
schema  = "schemas/obs.json"

[action]
kind    = "tool_call"       # or "text", "discrete", "mcp_invoke"
schema  = "schemas/act.json"
```

**Build path:** `benchflow.Env` base class (`reset / step / close / snapshot / restore`) → Docker backend first → observation/action schemas in `schemas/task/config.py` (optional in 0.3) → adapter shim (existing job-based tasks wrap as `JobEnv`) → reference integration: TRL-compatible loop on `tasks/rl-cartpole-mock`.

## §A7 · Harbor internalization — selective, not a fork

benchflow depends on `harbor==0.3.0` at runtime. The `internalize-harbor-experiments` branch vendored Task, Verifier, multi-agent schema, and harbor-registry compat.

**Policy** (xdotli, 2026-04-17): avoid massive code duplication with harbor. Importing from `harbor.*` is fine. Vendor only when we need to diverge.

**Vendoring criteria — vendor iff ≥1 is true:**
1. We need behavior upstream doesn't offer (e.g. structural `_harden()` as first line of `verify()`).
2. We need schema extensions upstream won't accept (`SkillRef`, multi-agent `[[agents]]`).
3. The upstream API is a constant churn risk and our runtime can't tolerate breakage.
4. Security-boundary fork (verifier hardening is the clearest example).

If none are true — import, don't vendor. Subclass or compose when a small extension is enough.

**Current branch audit:**

| Module | Status | Justified? |
|---|---|---|
| `Task / TaskConfig` | Vendored at `benchflow.schemas.task` | **Yes** — extended with `SkillRef`, multi-agent `[[agents]]`, env sidecar services. |
| `Verifier` | Vendored at `benchflow.verifier.Verifier` | **Yes** — `_harden()` runs first line; security-boundary fork. |
| `SkillRef`, multi-agent schema | New | N/A — adding. |
| `DockerEnvironment` | NOT vendored (issue #142) | **Don't vendor.** Import from `harbor.environments.docker`; subclass if needed. |
| `DaytonaEnvironment` | Imported from harbor | **Keep importing** — thin wrapper around Daytona's snapshot API. |
| Harbor registry compat | New glue (`harbor://`, `benchflow://` prefixes) | **Yes** — new code, not duplication. |
| Installed agent runners | Upstream; benchflow doesn't use | **Don't import, don't vendor** — our 5 ACP agents replace this tree. |

**Recommendation:** keep harbor as a pinned import-dep for 0.3. Full standalone (off `install_requires`) is a 0.3.x decision tied to closing issue #142.

## §Testing philosophy & regression loop

The "unit" for this framework is **not** a pytest function. It's **one benchmark task, one agent, one model, one scored result**. Coverage is measured in completed trajectories, not in lines hit.

**Three tiers:**

| Tier | What it is | When to run |
|---|---|---|
| **Unit** (framework-native) | 1 task × 1 agent × 1 model × scored result. | Every PR touching runtime, verifier, sandbox, or agent code. Sample ≤5 tasks, ~10-min CI budget. |
| **Smoke (parity)** | 50 tasks × 3 benchmarks vs 0.2.3 baseline. | Every feature-complete PR before merge; before any tag cut. |
| **Intervention** | Targeted probe: assert one behavior without a full task. E.g. "`emit_process_reward` reaches trajectory log" on a stubbed scene. | Future state — as corpus grows, replace some unit executions to keep CI cost down. Track ratio in release notes. |

**Regression loop per-feature:** change → Tier 1 (≤5 tasks) → Tier 2 (50-task parity) → merge. Any parity delta outside the gate is root-caused per-task, annotated, or reverted.

## §Open decisions

1. **Simulated-user placement** — benchflow core (preferred) or ClawsBench add-on.
2. **Models-as-skills granularity** — single LLM call wrap, or full sub-benchflow-session.
3. **A2A vs ACP coexistence** — ACP stays client↔agent (stdio); A2A added for peer agents. Confirm non-replacement.
4. **Mailbox vs generic async bus** — email-shaped only, or typed message queues.
5. **Registry hosting** — self-hosted HTTP, GitHub releases, or S3+signed URLs.
6. **Reward-rubric schema** — freeze as `{name, weight, score}` list, or open-ended JSON until three consumers.
7. **Parity gate threshold** — ±2 tasks net, or accept larger A2 deltas as intentional reward-hack hardening.
8. **Gemini ACP default mode** — latest evidence strongly favors `--acp --yolo`, but keep this technically open until the remaining mismatch audit and pending timeouts finish.
