# Multi-Agent Sandbox — Design Document

> "Moltbook in a box": multiple agents with different tool access in a shared
> environment, communicating via MCP.

## Background

### What we're building

A benchflow feature where N agents run in the same sandbox (or interconnected
containers), each with scoped tool access, communicating with each other via
MCP tools. This enables evaluation of:

1. **Multi-agent collaboration** — agents working together on complex tasks
   (e.g., coder + reviewer, researcher + writer)
2. **Agent-of-chaos scenarios** — adversarial agents testing robustness
   (from ClawsBench: 3 agents + shared repo + inter-agent communication)
3. **Skill-mediated workflows** — agents using skills to interact with
   shared infrastructure (Gmail, GCal, GDocs, Slack, etc.)

### Prior art

- **Harbor cookbook `simulated-user`**: FastMCP sidecar container exposes
  `ask_user` tool via streamable-http. Agent calls it like any MCP tool.
  Persona-driven, multi-turn state maintained server-side. Docker Compose
  networking between containers.
- **ClawsBench chaos simulation**: 3 agents in shared dev container, each
  with access to different repos, can talk to each other.
- **Jun/Taskbase "moltbook in a box"**: Agents with different tool access
  (Google Docs, repos, benchflow, Claude) chat with each other in Discord
  to complete complex tasks.

## Architecture

### Option A: Docker Compose with MCP Orchestrator (recommended)

Extends the harbor-cookbook simulated-user pattern to N agents.

```
docker-compose.yml (generated by benchflow)
│
├── orchestrator          # FastMCP server — routes messages between agents
│   ├── Exposes: ask_{agent_name}(msg) for each agent
│   ├── Maintains: message queues, conversation history
│   └── Logs: all inter-agent messages for trajectory analysis
│
├── agent-coder           # Claude Code via ACP
│   ├── MCP tools: [ask_reviewer, write_code, run_tests]
│   ├── Skills: coding-related skills
│   └── Instruction: "You are a coder. Build the feature."
│
├── agent-reviewer        # Codex via ACP
│   ├── MCP tools: [ask_coder, read_code, approve, request_changes]
│   ├── Skills: review-related skills
│   └── Instruction: "You are a code reviewer. Review PRs."
│
└── shared-workspace      # Shared volume mounted into all agent containers
    └── /workspace/       # All agents see the same filesystem
```

**Why Docker Compose**: Multi-container tasks are already supported by Harbor
(the cookbook has examples). benchflow just needs to generate the
docker-compose.yml and manage agent lifecycles.

### Option B: Single Container with Process Isolation

All agents run in the same container as separate processes, communicating
via local MCP (stdio or localhost HTTP).

Pros: Simpler, no Docker Compose needed, works on Daytona.
Cons: No filesystem isolation between agents, harder to scope permissions.

### Recommendation: Option A for full isolation, Option B as lightweight fallback.

## Task Format Extension

### `task.toml` for multi-agent tasks

```toml

[metadata]

[environment]

# Agent definitions
[[environment.agents]]

instruction = "prompts/coder.md"

[[environment.agents]]
name = "reviewer"
agent = "codex-acp"
model = "gpt-5.4"
instruction = "prompts/reviewer.md"

# MCP-based inter-agent communication
[[environment.mcp_servers]]
name = "orchestrator"
transport = "streamable-http"
url = "http://orchestrator:8000/mcp"

# Per-agent tool access (optional — default: all tools available)
[environment.tool_access]
coder = ["ask_reviewer", "write_file", "run_command"]
reviewer = ["ask_coder", "read_file", "approve", "request_changes"]

[agent]
timeout_sec = 600       # per-agent timeout
max_turns = 20          # max inter-agent message exchanges

[verifier]
timeout_sec = 120
```

### Orchestrator Server

The orchestrator is a FastMCP server that:

1. **Registers agents dynamically** from the task config
2. **Exposes `ask_{name}(message: str) -> str` tools** for each agent
3. **Routes messages** between agents using their ACP connections
4. **Maintains conversation history** for trajectory analysis
5. **Enforces turn limits** to prevent infinite loops
6. **Logs everything** for grading and debugging

```python
"""Multi-agent orchestrator — routes messages between agents via MCP.

Generated by benchflow for multi-agent sandbox tasks.
"""

from fastmcp import FastMCP
import httpx
import json
from pathlib import Path

mcp = FastMCP("orchestrator")

# Agent registry (populated from task config)
AGENTS = {}  # name -> {"url": "http://agent-X:port", "conversation": [...]}
MAX_TURNS = 20
turn_count = 0
transcript = []  # Full conversation log for trajectory analysis


def register_agent(name: str, url: str):
    """Register an agent endpoint."""
    AGENTS[name] = {"url": url, "conversation": []}


async def route_message(from_agent: str, to_agent: str, message: str) -> str:
    """Send a message from one agent to another."""
    global turn_count
    turn_count += 1

    if turn_count > MAX_TURNS:
        return f"[ORCHESTRATOR] Turn limit ({MAX_TURNS}) reached. Conversation ended."

    # Log the message
    entry = {
        "turn": turn_count,
        "from": from_agent,
        "to": to_agent,
        "message": message,
    }
    transcript.append(entry)
    AGENTS[to_agent]["conversation"].append({"role": from_agent, "content": message})

    # Forward via the receiving agent's ACP session
    # (In practice, this sends a user message to the agent's ACP session
    # and waits for the response)
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            f"{AGENTS[to_agent]['url']}/message",
            json={"from": from_agent, "message": message},
            timeout=120,
        )
        response = resp.json().get("response", "")

    # Log the response
    transcript.append({
        "turn": turn_count,
        "from": to_agent,
        "to": from_agent,
        "message": response,
    })

    return response


# Dynamic tool registration — one ask_{name} per agent
def create_ask_tool(target_name: str):
    @mcp.tool(name=f"ask_{target_name}")
    async def ask_agent(message: str) -> str:
        f"""Send a message to {target_name} and get their response."""
        # Determine caller from context (MCP session metadata)
        caller = "unknown"
        return await route_message(caller, target_name, message)
    return ask_agent


# Save transcript on shutdown
import atexit
def save_transcript():
    Path("/logs/orchestrator/transcript.json").write_text(
        json.dumps(transcript, indent=2)
    )
atexit.register(save_transcript)
```

## Execution Flow

```
1. benchflow reads task.toml, detects type = "multi-agent"
2. Generates docker-compose.yml:
   - orchestrator service (FastMCP + agent registry)
   - N agent services (one per [[environment.agents]])
   - shared-workspace volume
   - Internal network
3. Builds images, starts containers
4. Orchestrator registers all agents
5. For each agent:
   a. benchflow starts ACP session
   b. Agent receives its instruction.md
   c. Agent begins working, calling ask_{other_agent} as needed
6. Orchestrator routes messages, enforces turn limits
7. When all agents complete (or timeout):
   a. Save orchestrator transcript
   b. Run verifier against shared workspace
   c. Collect per-agent trajectories + orchestrator transcript
8. Cleanup containers
```

## Verifier for Multi-Agent Tasks

The verifier sees:
- `/workspace/` — shared workspace with all agent outputs
- `/logs/orchestrator/transcript.json` — full conversation history
- `/logs/agent-{name}/` — per-agent trajectories

Grading can check:
- **Task completion**: Did the agents produce the expected output?
- **Collaboration quality**: Did they communicate effectively?
- **Role adherence**: Did the coder code and the reviewer review?
- **Safety**: Did agents stay within their tool access boundaries?

## Dependencies

1. **Harbor internalization** — Docker Compose support requires moving
   beyond Harbor's single-container model. Currently blocked on kywch's
   harbor internalization plan.

2. **FastMCP** — Already used in harbor-cookbook. Need to add as optional
   dependency (`pip install benchflow[multi-agent]`).

3. **ACP multi-session** — benchflow currently manages one ACP session
   per SDK.run(). Multi-agent needs N concurrent sessions with message
   routing. The ACP client supports this (session IDs), but SDK.run()
   doesn't.

## Open Questions

1. **Synchronous vs async agent communication**: Harbor's simulated-user
   is synchronous (agent calls tool, blocks until response). For real
   collaboration, agents may need to work concurrently and check messages
   periodically. Start with synchronous (simpler), add async later.

2. **Agent identity in MCP context**: When agent-coder calls `ask_reviewer`,
   the orchestrator needs to know who's calling. FastMCP doesn't expose
   caller identity by default. Options: (a) separate MCP endpoints per
   agent, (b) include caller name in the tool args, (c) use MCP session
   metadata.

3. **Cost control**: N agents × M turns × model costs can get expensive fast.
   Need configurable turn limits and cost caps per task.

4. **Daytona support**: Docker Compose doesn't work on Daytona. Multi-agent
   on Daytona would need Option B (single container, process isolation).
   Defer to Phase 5.
