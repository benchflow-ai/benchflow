New knowledge

5. What we learned today

Verifier correctness was the biggest hidden variable

The strongest new technical knowledge is that a lot of earlier TB2 interpretation was not primarily about the model or sandbox; it was about legacy verifier compatibility. The critical fix was allowing hardened verification to correctly handle legacy pytest --ctrf style tasks by inferring the needed plugin load path.

Practical effect: some earlier zero-pass conclusions were simply false negatives.

Root mode mattered, but not alone

We now have a more refined answer than “TB2 just needs root.”

  • root mode helps on some system-heavy tasks
  • but root alone was not the main blocker
  • verifier correctness + model choice + task fit all matter substantially

Model differences became visible again once the harness got cleaner

The strongest current example is pytorch-model-recovery:

  • Opus slices timed out on it
  • Gemini passed it

This is important because it means benchmark outcome is not simply monotonic with “larger model = better.” Tool-use style and task fit matter.

Harbor parity has to be reusable, not improvised

We learned not to over-read one-off Harbor probes. The durable Harbor line should be preselected and then reused. The agreed future default is:

Harbor agent: gemini-cli
Model: gemini-3.1-flash-lite-preview
Backend: Daytona
SkillsBench: 10 tasks
Terminal Bench 2.0: 89 tasks

Main operational knowledge from today

The reusable integration-test shape is now clear. That is arguably the single most important operational takeaway from this whole thread.
