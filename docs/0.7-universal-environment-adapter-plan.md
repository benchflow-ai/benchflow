# BenchFlow 0.7: universal environment adapter plan

## Next-session goal

The next 0.7 session should enable browser-use and computer-use evals to run
through BenchFlow while preserving the existing architecture separation:

- sandbox provider: where the world runs, including Cua for desktop/computer-use
  environments
- environment adapter: the browser or desktop world the agent acts in
- agent adapter: the decision loop, such as Browser Use, Stagehand, ACP agents,
  or a CUA model loop
- benchmark adapter: the importer/reference layer for external eval suites

The verification is intentionally simple: a BenchFlow run of the adapted evals
must work and be comparable to the original runner on the same selected tasks.
For each adapter, compare pass/fail or reward, trace completeness, timing,
artifact shape, cleanup behavior, and unsupported-task reporting.

## Product goal

BenchFlow 0.7 should make benchmark onboarding feel like this:

1. Discover a benchmark source.
2. Import or reference its tasks through one adapter contract.
3. Choose an agent loop.
4. Choose a sandbox provider.
5. Run a small verified slice.
6. Scale only after artifacts, cleanup, and parity checks pass.

When a new browser-use or computer-use benchmark appears, BenchFlow can adapt to
it without rewriting the rollout kernel, agent registry, or sandbox providers.

## From 0.6 to 0.7

0.6 is the task-standard release: `task.md`, native verifier/oracle packaging,
runtime-capability checks, and the core single-benchmark run path. It gives
BenchFlow a stable unit of work.

0.7 should be the environment-adapter and loopcraft release. The unit of work
stays the same, but the adoption process becomes repeatable: discover a foreign
benchmark, recognize its task shape, map provider/environment/agent
requirements, run the original runner, run BenchFlow, compare artifacts, and
only then scale.

The important shift is from "an engineer writes an adapter" to "BenchFlow runs
an adapter-adoption loop." A benchmark adapter is not done when it imports a
task; it is done when the loop produces original-vs-BenchFlow parity evidence
and a provider-honest unsupported report for the rest.

## Current evidence status

The current 0.7 branch is past scaffolding but not yet at a release-quality
evidence gate.

Working now:

- raw Browser Use-shaped task dirs can be materialized through the inbound
  adapter seam and run through `bench eval create`
- `bench environment check/create --json` now reports the environment-adapter
  plane separately from the benchmark adapter and sandbox provider; Browser
  Use tasks report `environment_adapter.name=browser`, and computer-use /
  use-computer cookbook tasks report `environment_adapter.name=desktop`.
  Desktop reports distinguish verified local Cua from cloud Cua that still
  requires a live runtime probe.
- Browser Use CLI, Browser Use Agent, and Stagehand Agent slices run through
  BenchFlow on the local browser smoke and produce parity reports
- browser/computer-use parity scripts now emit a scrubbed
  `adoption_report.json` plus `loop_state.json` next to
  `parity_experiment.json`, so the adoption loop has a durable manifest for
  architecture planes, parity counts, artifact counts, timing coverage,
  cleanup, replay commands, role status, and next queue items without retaining
  raw screenshots
- `bench agent verify --json` now surfaces an optional `adoption_report.json`
  sidecar when present, so loop controllers can consume the parity verdict and
  scrubbed architecture/artifact manifest from one existing CLI command
- `bench agent verify --require-adoption-report --loop-report-out <path>
  --json` now acts as the strict adapter-adoption scale gate: parity must pass
  and `adoption_report.json` must prove planes, reward delta,
  trace/screenshot artifacts, eval summary, timing, cleanup criterion, and Cua
  provider mode when relevant. The same gate now also requires
  `loop_state.json` to validate and be at least `review-ready`, so a benchmark
  cannot scale from a transient parity report without durable controller state.
- `loop_state.json` is now the resumable loopcraft artifact. It records source
  identity/revision, selected tasks, exact commands, parity/adoption artifacts,
  role status for scout/builder/original-runner/BenchFlow-runner/verifier/
  auditor/reviewer/queue, cleanup, unsupported summary, and next chunks. The
  machine gate can be scale-ready while this controller state remains
  `review-ready` until a reviewer role explicitly passes.
- `bench environment list/cleanup --sandbox docker --json` now exposes the
  same provider-cleanup surface for Docker that the browser-use smoke relies
  on. It is scoped to BenchFlow-owned containers/networks with
  `benchflow.owned=true`, so the adoption loop can inspect and clear stale
  Docker resources before rerunning parity.
- browser-oriented shims now share `benchflow.environment.browser_runtime` for
  localhost fixture serving, readiness checks, scrubbed snapshot metadata, and
  prompt rewriting; that helper is bundled into the sandbox install payload for
  Browser Use smoke, Browser Use CLI, Browser Use Agent, and Stagehand so the
  split survives real sandbox dogfood runs
- one official Browser Use encrypted-task slice can be imported into `/tmp` and
  run through BenchFlow with the real Browser Use Agent and `llm-judge`
- two official Stagehand eval tasks (`agent/sign_in`, `agent/steam_games`) can
  be imported from a Stagehand checkout into `/tmp` as `stagehand-task.json`,
  validated through `bench tasks check --level runtime-capability --sandbox
  docker`, run through BenchFlow with the real Stagehand Agent loop at reward
  `1.0`, and compared against the original Stagehand runner on the same task.
  The repeatable parity driver now writes `parity_experiment.json` plus
  `adoption_report.json`; it prefers Stagehand's built `dist/esm/tasks` tree
  and falls back to source tasks with `tsx`.
- Cua local can run desktop/computer-use smoke tasks with shell, screenshot,
  file transfer, timing, trajectory, cleanup, and original-vs-BenchFlow parity
- use-computer cookbook OSWorld and supported CUA-Gym slices can run on local
  Cua through the existing `tasks`, `environment`, and `eval` CLI nouns
- iOSWorld is recognized and returns a structured unsupported report requiring
  a `macos-ios-simulator` provider instead of failing as an unknown task format

Not release-grade yet:

- the browser world has a shared runtime helper and parity-gated readiness
  snapshots now, and browser-oriented shims now emit artifacts through
  `BrowserRuntimeSession` with schema `benchflow.browser-runtime-trace.v1`.
  Remaining release work is a full browser runtime provider that owns external
  browser launch/session lifecycle beyond the current Browser Use/Stagehand
  agent-owned sessions.
- Stagehand now has two official same-task original-vs-BenchFlow parity slices,
  a repeatable driver, a strict adoption-loop gate pass on fresh upstream
  evidence, and an all-task support inventory. Release evidence still needs
  coverage beyond deterministic URL-check tasks and a mapped verifier /
  expected-answer reward contract.
- Cua cloud is not healthy evidence yet: with the saved handoff key, VM
  creation/list/delete works, but the command endpoint returned HTTP 404 until
  the bounded runtime probe reported `not-ready`; the JSON report now names the
  failed runtime capabilities instead of leaving adapters to infer the failure
- Cua cloud failure-mode cleanup is now live-verified for the probe path with
  final VM count `0`; SDK background readiness failures are captured in
  `runtime_probe.background_errors` so JSON output stays structured
- Cua cloud probe diagnostics now include SDK/version metadata, request
  metadata, and a normalized failure class. The current live failure on
  2026-06-12 is `cloud-computer-server-cmd-404` with `cua-sandbox==0.1.16`,
  `linux_kind=vm`, and cleanup/list evidence proving `0` remaining VMs.
- scrubbed evidence manifests now live beside the browser/computer-use/iOSWorld
  smoke fixtures and `tests/integration/check_adapter_evidence.py` enforces
  their parity, artifact, eval summary, cleanup, and unsupported-reporting
  fields; Cua slices must declare `sandbox_provider_mode` as `local` or
  `cloud-probed` so local parity cannot be mistaken for cloud runtime support
- full raw live run artifacts, screenshots, and provider logs are still
  regenerated outside the repo; `adoption_report.json` now persists the
  scrubbed manifest layer, but release evidence should still retain an official
  report bundle for the selected browser and desktop slices

## Research summary

Loopcraft framing: recent agent tooling discussion argues that the leverage is
moving from prompting agents directly to designing loops that prompt agents,
stacking loops upward for leverage and downward for reliability:

- https://www.latent.space/p/ainews-loopcraft-the-art-of-stacking

Goal-flight is a concrete long-running controller pattern: durable goal files,
queue chunks, worker dispatch, capacity/rate-limit awareness, executor
self-review, independent chunk review, milestone review, RAG corpus slices, and
resume notes:

- https://github.com/simonrowland/goal-flight

Stagehand separates evals by capability and benchmark shape. Its docs describe
act, extract, observe, combination, experimental, agent, and external benchmark
evals. Its CLI can run categories or external benchmarks like WebBench, GAIA,
WebVoyager, OnlineMind2Web, and OSWorld, with knobs for environment, trials,
concurrency, model, provider, and API mode:

- https://docs.stagehand.dev/v3/basics/evals
- https://www.stagehand.dev/evals

Browser Use's benchmark repo separates benchmark tasks, browser providers, and
framework adapters. It has a `frameworks/` directory for agent/framework loops
and a `browsers/` directory for browser infrastructure. Its framework runner
passes normalized environment variables into subprocess adapters and writes
results plus task traces:

- https://github.com/browser-use/benchmark

The use-computer cookbook adapter directory converts external sources into task
directories. It has separate adapters for hand-written tasks, collected tasks,
CUA-Gym, macOSWorld, and WindowsAgentArena. The pattern is "load source, filter
unsupported cases, materialize task dirs, carry verifier/grading assets":

- https://github.com/josancamon19/use-computer-cookbook/tree/main/src/runner/adapters

iOSWorld is a native iOS Simulator benchmark. Its repository ships 26 SwiftUI
apps, 133 tasks, rubrics, local/parallel runners, optional vision plus
accessibility XML observations, Appium/XCUITest control, and local Mac or EC2
Mac setup paths. It requires macOS, Xcode 26+, an iOS 26 simulator runtime,
Python, Node.js, Appium, and a judge key for the default scoring path:

- https://iosworld.io/
- https://github.com/ljang0/iOSWorld

Cua's Sandbox SDK is the right sandbox substrate for desktop/computer-use
environments. It exposes sandbox creation plus shell, screenshot, mouse,
keyboard, clipboard, and display capabilities:

- https://cua.ai/docs/cua/guide/get-started/using-sandbox-sdk

The Cua local dogfood exposed three provider gaps that are now part of the
adapter contract:

- root-owned runtime directories such as `/logs/agent` must be initialized as
  root, not as the default desktop user
- non-Docker providers must honor Dockerfile `WORKDIR` when task config does
  not set `environment.workdir`
- the verifier must run main-service test scripts from the same resolved
  workspace as the agent/oracle

## Architecture

0.7 should make four planes explicit.

### Sandbox provider

Owns infrastructure lifecycle and low-level operations:

- `create`
- `connect`
- `exec`
- `upload`
- `download`
- `ports`
- `logs`
- `delete`

Examples: Docker, Daytona, Modal, Cua.

### Environment adapter

Owns the world the agent acts in:

- `materialize(task)`
- `start(sandbox)`
- `check()`
- `reset()`
- `snapshot()`
- `restore()`
- `observe()`
- `artifacts()`
- `cleanup()`

Examples: repo task, browser task, desktop task, hosted Verifiers env,
manifest-backed benchmark env.

### Agent adapter

Owns the decision loop:

- `prepare(environment_handle)`
- `run(prompt, tools)`
- `record_step`
- `stop`
- `usage`
- `trajectory`

Examples: ACP code agents, Browser Use, Stagehand, CUA model loop, OpenHands.

### Benchmark adapter

Owns foreign benchmark ingestion:

- `discover(source)`
- `select(filters)`
- `convert_or_reference(task)`
- `materialize_task_dir`
- `attach_verifier`
- `parity_sample`

Examples: BU Bench, CUA-Gym, WindowsAgentArena, OSWorld, macOSWorld,
Stagehand external benchmarks, iOSWorld.

## Loop engineering

BenchFlow already has the primitive needed for multi-agent build and
verification loops: `Scene`, `Role`, and `Turn` run multiple agents against the
same sandbox and preserve handoffs through explicit artifacts. For 0.7, use
that primitive as the adapter-factory loop rather than adding a separate
orchestration product surface.

The 0.7 loop stack should look like this:

| Loop | Purpose | BenchFlow artifact |
|---|---|---|
| Goal loop | Maintain north star, scope, assumptions, blockers | `docs/0.7-*` plan plus private context handoff |
| Adapter loop | Recognize/import one benchmark slice | inbound adapter + support report |
| Provider loop | Prove sandbox/environment readiness | `bench environment check/list/cleanup` evidence |
| Agent loop | Run Browser Use, Stagehand, CUA, ACP, or oracle | trajectory + usage + artifacts |
| Parity loop | Compare original runner vs BenchFlow | parity report: reward, trace, timing, artifacts |
| Review loop | Attack the adapter and evidence | self-review, independent review, regression tests |
| Scale loop | Run broader matrix after parity passes | batch health, failure taxonomy, cleanup report |

One adapter-adoption flight:

1. Scout role: inspect the upstream benchmark repo, identify task schema,
   runner entry points, environment requirements, verifier shape, artifacts,
   credentials, and unsupported cases.
2. Builder role: implement the smallest adapter slice through the inbound
   adapter seam and existing CLI resource verbs.
3. Original-runner role: run the benchmark's own selected task runner and
   preserve pass/fail or reward, timing, traces, screenshots, and artifacts.
4. BenchFlow-runner role: run the same selected tasks through
   `bench tasks check`, `bench environment create/check`, and
   `bench eval create`.
5. Verifier role: compare original-vs-BenchFlow reward, trace completeness,
   timing, artifact shape, cleanup behavior, and unsupported-task reporting.
6. Auditor role: run provider leak checks, public artifact denylist scans,
   dependency/license checks, and ensure unsupported reasons are actionable.
7. Reviewer role: run concern-diverse review on the adapter diff and the parity
   evidence. Any confirmed issue becomes a regression test or a documented
   unsupported reason.
8. Queue role: append follow-on coverage chunks such as `open` setup support,
   mock app support, Mac/iOS provider support, or additional reward dependency
   mappings.

The loop only graduates an adapter from recognized to runnable when the
original runner and BenchFlow runner agree on the same selected tasks. If a
provider capability is missing, the adapter should return a structured
unsupported report first, so a batch run can separate provider gaps from agent
failures.

Loopcraft contract for multi-agent work:

- The controller owns a durable `goal.md`, `adoption_report.json`,
  `loop_state.json`, and `loop-report.json`; workers do not pass critical
  state only through chat history.
- Each role writes one replayable artifact: source inventory, adapter diff,
  original-runner result, BenchFlow result, parity report, cleanup report, or
  review report.
- Every success claim names the exact reproduction command and artifact path;
  `bench agent verify --require-adoption-report --loop-report-out <path>
  --json` is the machine-readable checkpoint before scale.
- A reviewer role can fail the loop by finding missing parity, missing
  screenshot/trace/timing coverage, leaked provider resources, ambiguous
  unsupported cases, or public secret/competitor leakage.
- The scale loop is locked until the one-task parity loop passes on both the
  original runner and BenchFlow runner.

Implementation shape:

- Keep the public CLI on existing nouns: `bench tasks`, `bench environment`,
  and `bench eval`.
- Store loop templates as scenes/configs, not as a new top-level command group.
- Record every loop pass as artifacts: upstream source revision, selected task
  ids, original command, BenchFlow command, parity table, unsupported summary,
  cleanup summary, and denylist scan.
- Use high-concurrency providers only after the one-task parity loop passes.
- Require provider-honest blockers for new native surfaces such as iOS
  Simulator, macOS desktop, Windows desktop, hosted browsers, and mock web apps.
- Keep loop state file-backed so compaction or a new coding session can resume:
  goal prompt, queue, selected task ids, exact commands, result paths, review
  outputs, and remaining blockers. The implemented
  `benchflow.adapter-adoption-loop-state.v1` schema is intentionally a data
  artifact, not a new public CLI command.
- Treat missing/stalled review as inconclusive, not clean.
- Keep the controller out of hot execution where possible: workers implement,
  original runners compare, reviewers attack, and the controller escalates only
  product, auth, cost, or provider-capability decisions.

Verification:

- 3-task x 2-agent x 2-trial dogfood matrix.
- All trajectories complete and auditable.
- No leaked secrets or unpublished benchmark plaintext.
- Provider cleanup report is empty after run.
- `bench environment cleanup --sandbox cua --dry-run --json` emits parseable
  cleanup reports with `found`, `matched`, `skipped`, `deleted`, and
  `candidates`, so a dogfood loop can assert no BenchFlow-owned Cua resources
  would be deleted after a run.
- `bench environment check ... --sandbox cua --probe-runtime --json` starts a
  bounded Cua provider probe when live runtime evidence is needed. The JSON
  report includes shell, upload/download file transfer, screenshot, dimensions,
  display metadata, and cleanup status without logging screenshots, display
  URLs, or credentials. Display URL support is reported as a capability result;
  local Cua currently reports it unsupported while still passing the required
  runtime checks. Cloud failures also include SDK/version metadata, request
  metadata, `error_type`, and a normalized `failure_class` such as
  `cloud-computer-server-cmd-404`.
- `bench tasks check ... --json` emits parseable adapter-adoption records:
  supported tasks return `status=valid`; recognized unsupported tasks return
  `status=unsupported-adapter-task` with `adapter`, `dataset`, `reason`, and
  structured `details`.
- `parity_test.py --parity-out <path>` now records an adapter artifact
  manifest inside `parity_experiment.json`. `bench agent verify` scores it as
  the `artifact-manifest` criterion, so missing traces, screenshots, screenshot
  methods, or trajectory fields fail the parity gate before scale.
- `bench agent verify ... --json` emits a parseable parity verdict with
  conversion counts/disagreements, reward samples/deltas, optional round-trip
  status, and issue-draft metadata. The adapter-adoption loop can now consume
  task check, environment readiness, eval run, cleanup, and parity verdicts as
  JSON.
- `bench agent verify ... --require-adoption-report --loop-report-out <path>
  --json` promotes the optional sidecar into a strict scale gate. It fails when
  the scrubbed adoption report is missing, trace/screenshot artifacts are thin,
  eval summary/timing evidence is absent, reward deltas exceed tolerance, or a
  Cua slice omits `sandbox_provider_mode`.
- `bench eval create ... --json` emits a parseable run report. Local/repo runs
  include the persisted `summary.json` payload plus score-view counts, so the
  loop can compare BenchFlow pass/fail, timing, trace/artifact summaries, and
  errors against the original runner without scraping console output.
- `parity_test.py` now runs BenchFlow with `bench eval create --json`, stores
  the resulting eval report in `summary.benchflow_eval`, and writes
  `loop_state.json` beside `parity_experiment.json` and `adoption_report.json`.
  The parity builder scores the eval report as the `eval-run-summary`
  criterion. This catches missing `summary.json`, errored evals, missing
  timing, and trace-thin runs before scale, while the loop state lets another
  agent resume the scout/build/original-runner/BenchFlow-runner/verifier/
  auditor/reviewer queue.
- Browser fixture parity manifests now require the environment adapter
  readiness snapshot: `environment.readiness.status=ready` plus a
  `content_sha256` for the served local fixture. This keeps browser-world
  readiness visible separately from the agent's final answer and screenshot

## Dogfood matrix

| Layer | Smoke | Required before scale |
|---|---|---|
| Task schema | `bench tasks check` | structural + runtime-capability pass |
| Environment object | `bench environment create` | dry-run diff + clean create |
| Provider readiness | `bench environment check` | auth/deps/readiness clean error |
| Browser env | static local page | screenshot + verifier artifact |
| Desktop env | Cua local desktop smoke | shell + screenshot + cleanup |
| Agent loop | one fixture task | trajectory + usage + final output |
| Benchmark adapter | cookbook OSWorld/CUA-Gym smokes + Python-reward CUA-Gym slices | parity sample + unsupported report |
| Cleanup | provider list before/after | no leaked VM/container/session |

## Risks

- Cua cloud can report VM `running` while the command endpoint still returns
  HTTP 404. Treat this as a provider readiness failure, not an agent failure.
- Browser benchmarks may contain encrypted or policy-sensitive task text. Never
  commit decrypted tasks or run data.
- Stagehand/Browser Use public benchmark adapters may be partly scaffolded or
  changing quickly. Prefer their official docs and runnable CLI over stale
  framework stubs.
- `bench agent` is already overloaded between solver agents and benchmark
  adoption workflows. 0.7 should avoid adding more meanings to that group.
