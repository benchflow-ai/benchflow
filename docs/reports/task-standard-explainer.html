Understanding the BenchFlow task standard after dogfooding it

The new standard is best understood as a package contract: the task asks, the verifier judges, the oracle proves solvability, and the runtime view decides what can safely execute on a chosen backend.

TL;DR

What changed

`task.md` is not the whole product

It is the native authoring entrypoint. The product surface is the full package: `task.md`, `verifier/`, `oracle/`, environment, and evidence.

Big insight

`verifier/` is a peer package

`verifier/test.sh` remains the minimum script path, but `verifier/verifier.md` should own rubrics, Reward Kit criteria, judge agents, reward details, hidden inputs, and calibration.

Next work

Build the runtime view first

`TaskRuntimeView` and `VerifierDocument` are the next slice. Without them, today’s checker validates shape but ignores the verifier package semantics.

Mental Model

The old way to think about task packages was “instruction plus tests.” That breaks down for Harbor forks, Pier/DeepSWE, FrontierSWE, AgentBeats-style assessors, Reward Kit, and ORS-style reward streams. The standard that survived dogfooding is more like this:

Task asks. The root package describes what the solver sees: config, prompt, roles, scenes, user loop intent, and compatibility metadata.
Verifier judges. The verifier package describes hidden inputs, rubrics, reward strategies, judge agents, output contracts, and anti-cheat/calibration evidence.

Dogfood Tasks

Four real BenchFlow feature tasks were packaged using the standard. Each one has `task.md`, `verifier/verifier.md`, `verifier/test.sh`, a rubric, `oracle/solve.md`, and an environment Dockerfile.

What It Proved

The naming held up

`solution -> oracle` and `tests -> verifier` made the packages easier to reason about. Oracle reads as reference behavior; verifier reads as the whole judgment contract.

`benchflow:` is doing useful containment

Draft semantics like nudges, verifier package metadata, reward strategy, and compatibility losses did not need to become root task keys.

Real tasks need verifier intent

Bare `test.sh` is not enough for features involving Reward Kit, rich reward details, judge agents, export losses, or calibration evidence.

Structural validity is not runtime validity

All four packages pass today’s `bench tasks check`, even though their verifier package semantics are future-facing. That is the useful discomfort.

Thin Spots

  • `VerifierDocument` is missing. Current code parses root `task.md`, but it does not parse or enforce `verifier/verifier.md`.
  • Capability checks need a second validation level. Structural validation passes too much by design. We need sandbox-aware validation for parsed-but-unsupported semantics.
  • Future acceptance scripts need a status convention. The dogfood verifier scripts represent desired acceptance tests, not currently runnable task verification.
  • Compatibility exports need a typed loss report. The standard can describe losses, but the implementation should emit a concrete JSON artifact.

Next Slice

Implement `TaskRuntimeView` and `VerifierDocument` before expanding more syntax. The dogfood pass shows the standard is expressive enough; the risk now is parsing attractive metadata that the runtime silently ignores.

Implementation order

1. TaskRuntimeView / TaskPackage
2. runtime_capabilities.py
3. VerifierDocument
4. reward.json + reward-details preservation
5. Harbor/Pier export loss reports
6. prompt/user runtime compilation
