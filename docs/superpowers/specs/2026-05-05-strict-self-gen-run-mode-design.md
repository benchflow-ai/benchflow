# Strict Self-Generated Skill Run Mode Design

## Context

PR #233 currently implements self-generated skills as two scenes inside one
BenchFlow trial and one sandbox. That does not match the required semantics.
The strict behavior must be:

1. Run a creator phase like a normal no-skill run, except with the
   `skill-creator` skill directory mounted and a skill-generation prompt.
2. Bundle only the skills generated by that phase.
3. Run the real task in a fresh isolated environment like normal with-skills
   mode, using only the generated skills.

## Requirements

- `self-gen` must preserve the same task configuration as the requested run:
  agent, model, agent environment, sandbox backend, sandbox user, locked paths,
  setup timeout, context root, and other run-level settings.
- The creator phase must run in its own isolated environment.
- The creator phase must skip the verifier.
- The creator phase must not use task-bundled original skills.
- The creator phase must mount only the configured skill creator skill pack as
  its skills directory.
- The creator phase may generate more than one skill.
- Creator success requires at least one generated skill directory containing
  `SKILL.md`.
- The solver phase must run in a separate fresh isolated environment.
- The solver phase must run the normal task instruction and verifier.
- The solver phase must behave exactly like existing with-skills mode, with
  `skills_dir` set to the host directory containing only the generated skills.
- The solver phase must not mount the `skill-creator` pack.
- The solver phase must not mount task-bundled original skills.
- The final reported benchmark result must come from the solver phase only.

## Architecture

Replace the current scene-based self-gen implementation with a small runtime
orchestrator around two normal `Trial` executions.

The orchestrator should live at the SDK/job layer where single-task execution
is already assembled. For `skill_mode != "self-gen"`, behavior stays unchanged.
For `skill_mode == "self-gen"`, the orchestrator runs:

1. Creator trial.
2. Generated-skill export and validation.
3. Solver trial.

This keeps `Trial` focused on one isolated run lifecycle and avoids special
scene semantics for a feature whose main requirement is environment isolation.

## Creator Trial

The creator trial uses the same task path and run configuration as the
requested task run, with these overrides:

- `skill_mode="default"` or no special mode, so it uses the normal single-run
  path.
- `skills_dir=<resolved skill_creator_dir>`.
- task-bundled original skills disabled.
- prompts replaced with one creator prompt that instructs the agent to inspect
  `/app/instruction.md` and the task environment, then create one or more
  Anthropic-standard skills under a deterministic output root.
- verifier skipped.

The official vendored Anthropic `skill-creator` remains the default
`skill_creator_dir`. User overrides through `--skill-creator-dir` and
`BENCHFLOW_SKILL_CREATOR_DIR` remain supported.

## Generated Skill Export

After creator execution, BenchFlow copies the generated skill root from the
creator sandbox into a host artifact directory under the job/trial output.

The exported host directory is considered valid if it contains at least one
immediate child directory with `SKILL.md`. BenchFlow should fail self-gen early
with a clear error if no generated skill is found.

The validation intentionally checks only `SKILL.md` existence for this runtime
path. It should not reject additional Anthropic-standard resources such as
`scripts/`, `references/`, `assets/`, `examples/`, or multiple generated skill
directories.

## Solver Trial

The solver trial uses the same task path and run configuration as the requested
task run, with these overrides:

- `skill_mode="default"` or no special mode.
- `skills_dir=<host exported generated skills dir>`.
- task-bundled original skills disabled.
- normal task prompt resolution.
- verifier enabled.

Because this is a fresh `Trial`, it gets a fresh sandbox/environment and cannot
see the creator environment, creator process state, or mounted `skill-creator`
pack. The only bridge between phases is the host-exported generated skills
directory.

## CLI, SDK, Job, and YAML Surface

Keep the public options introduced in PR #233:

- `--skill-mode self-gen` / `--mode self-gen`
- `--skill-creator-dir`
- `--self-gen-no-internet`
- YAML fields `skill_mode`, `skill_creator_dir`, and `self_gen_no_internet`

`--skills-dir` should not be used as the solver skills input in self-gen mode.
The solver skills directory is always the generated skill export. Any task
original skills are ignored for both phases.

## Error Handling

- If the creator trial fails before exporting skills, return a final result
  with a clear self-gen creation error.
- If no generated child directory with `SKILL.md` exists, return a clear
  validation error.
- If exporting generated skills fails, return a clear export error.
- Solver verifier errors and rewards remain normal BenchFlow solver results.

## Testing

Add focused tests for:

- self-gen orchestration constructs two separate trial executions.
- creator trial receives same agent/model/env/sandbox config as the requested
  run.
- creator trial mounts only skill creator skills and skips task original skills.
- creator trial skips verifier.
- generated skill export accepts one or more child directories with `SKILL.md`.
- export validation rejects an empty generated skills root.
- solver trial receives a fresh config with `skills_dir` set to the exported
  generated skills directory.
- solver trial does not receive the skill creator directory.
- solver trial disables task original skills.
- CLI/YAML/Job/SDK paths all reach the same strict self-gen orchestration.

## Out of Scope

- Evaluating the semantic quality of generated skills beyond checking for
  `SKILL.md`.
- Running the creator phase verifier.
- Sharing a sandbox between creator and solver phases.
- Injecting task original bundled skills into either self-gen phase.
